ВВЕДЕНИЕ

В отчете представлены результаты выполнения первых двух лабораторных работ по дисциплине «SQL и получение данных». В рамках проекта реализован сквозной процесс обработки данных: генерация исходного набора, загрузка в PostgreSQL, очистка и типизация данных средствами SQL, построение витрины данных, а также перенос витрины в целевую таблицу MySQL.

Цель работы — разработать практический ETL-конвейер и структуру хранилища данных, соответствующие требованиям лабораторных заданий текущего года.

Практическая ценность выполненной работы заключается в формировании воспроизводимого процесса, который можно запускать повторно без изменения логики обработки. Такой подход обеспечивает сопоставимость результатов между запусками и упрощает проверку корректности данных на каждом этапе. Дополнительно реализована формальная структура проекта, разделяющая код приложения и SQL-объекты, что соответствует инженерной практике сопровождения аналитических систем.

1 Описание предметной области

1.1 Бизнес-контекст

Предметная область — обработка заказов условного интернет-магазина. Для аналитики требуется получать корректные данные о заказах, категориях, статусах и источниках.

В операционных системах интернет-торговли данные поступают из нескольких каналов: веб-сайт, мобильное приложение, CRM-система. На практике это часто приводит к неоднородности форматов, пропускам, ошибкам в статусах и числовых полях. Вследствие этого аналитические отчеты могут давать искаженные результаты, если использовать данные напрямую из источников.

В рамках проекта моделируется типичная задача аналитического контура: сформировать устойчивый слой данных для отчетности, в котором запись о заказе имеет согласованные атрибуты, корректные типы и проверенные бизнес-ограничения. Таким образом, реализованный процесс отражает базовую роль инженерии данных в бизнесе — преобразование «сырого» потока событий в надежный набор показателей для принятия решений.

1.2 Тип данных

Обрабатываются транзакционные данные о заказах: дата, номер заказа, клиент, город, категория товара, статус, количество, цена, скидка, источник данных.

На этапе генерации все значения подготавливаются как строковые, что имитирует типичную выгрузку из внешнего источника без строгой схемы. Далее данные проходят этап приведения к целевым типам: даты, целые и десятичные числа, а также категориальные признаки. Такое разделение позволяет явно продемонстрировать этап Transform как обязательную часть ETL-процесса.

1.3 Назначение витрины данных

Витрина v_dm_task предназначена для аналитических запросов: данные уже очищены, структурированы и содержат внешние ключи на справочники.

2 Общее описание системы

2.1 Назначение и границы системы

Система реализует пакетную обработку данных и ориентирована на учебный сценарий выполнения ЛР1 и ЛР2. Пользовательский интерфейс не реализуется; взаимодействие выполняется через Python-скрипты, SQL и DBeaver.

Границы системы включают генерацию данных, загрузку в PostgreSQL, построение DWH-структуры и перенос итоговой витрины в MySQL. Вне границ находятся визуальная BI-аналитика, пользовательский frontend, потоковая обработка и внешние API-интеграции. Таким образом, акцент сделан на корректности вычислительной и SQL-логики, а не на интерфейсных аспектах.

2.2 Входные данные

Входные данные генерируются синтетически в Python-модуле get_dataset.py. Набор содержит преднамеренные аномалии (некорректные даты, статусы, цены, пустые значения и др.), что позволяет продемонстрировать очистку средствами SQL.

2.3 Исходящие данные

  1. Очищенная и типизированная таблица s_sql_dds.t_sql_source_structured.
  2. Фактовая таблица DWH s_sql_dds.t_dm_task.
  3. Витрина s_sql_dds.v_dm_task.
  4. Целевая таблица MySQL sql_dm.t_dm_task с данными из витрины.

2.4 Используемые технологии и программные средства

  • Python 3.11
  • PostgreSQL 16
  • MySQL 8.0
  • SQL / PLpgSQL
  • Docker + Docker Compose
  • DBeaver

Каждый инструмент выбран для конкретной роли: Python управляет последовательностью шагов и интеграцией между СУБД, PostgreSQL отвечает за преобразование и хранение структурированных данных, MySQL используется как целевая внешняя система для перекладки витрины, SQL фиксирует бизнес-правила преобразования в явном и проверяемом виде.

3 Архитектура проекта

3.1 Общая схема обработки данных

get_dataset() -> load_data_to_db() -> fn_etl_data_load() -> fn_dm_data_load() -> v_dm_task -> transfer_dm_to_mysql()

Последовательность построена так, чтобы каждый шаг имел однозначный вход и выход. На Python-этапах выполняется orchestration (управление порядком вызовов), а на SQL-этапах — детерминированная обработка и фильтрация данных. Это уменьшает связанность между слоями и облегчает отладку: при ошибке можно локализовать проблемный узел по этапу конвейера.

3.2 Слои данных и их назначение

  • Raw слой: t_sql_source_unstructured (все поля как строковые, «сырые» данные).
  • Structured слой: t_sql_source_structured (очищенные и типизированные данные).
  • DWH слой: справочники t_dim_* и фактовая таблица t_dm_task.
  • Data Mart: представление v_dm_task.
  • Target: MySQL-таблица t_dm_task.

3.3 Взаимодействие компонентов системы

Python-слой управляет запуском этапов. SQL-слой выполняет очистку и построение DWH. DBeaver используется для ручной верификации результатов. Docker объединяет все компоненты в единое переносимое окружение.

Ключевой принцип взаимодействия компонентов — разделение ответственности. Приложение не дублирует SQL-правила очистки, а только инициирует вызов хранимых функций с периодом расчета. В свою очередь, SQL-функции не зависят от интерфейса запуска и могут быть вызваны как из приложения, так и напрямую из SQL-клиента. Такой подход позволяет использовать единый вычислительный контур как в автоматическом, так и в ручном режиме проверки.

4 Описание лабораторных работ

4.1 Лабораторная работа 1. ETL-процесс

4.1.1 Входные данные

Синтетически сгенерированный набор из 10 полей, содержащий корректные записи и аномалии.

4.1.2 Логика преобразований и очистки данных

  • Проверка и парсинг даты и числовых полей через регулярные выражения.
  • Нормализация строк (trim/lower).
  • Фильтрация недопустимых статусов и диапазонов значений.
  • Удаление дублей по ключу (дата, заказ).

Логика очистки реализована полностью на стороне SQL, что соответствует требованиям лабораторной работы. Функция принимает период start_date и end_date, очищает только соответствующий диапазон целевой таблицы и повторно загружает в него валидные записи. Такой механизм поддерживает повторный расчет без накопления дубликатов и обеспечивает управляемость перезапусков.

4.2 Лабораторная работа 2. Хранилище данных и витрина

4.2.1 Модель данных хранилища

Реализована модель «звезда»: в центре таблица фактов t_dm_task, по краям справочники t_dim_customer, t_dim_city, t_dim_product_category, t_dim_status, t_dim_source_system.

4.2.2 Справочники и фактовая таблица

Справочники заполняются из категориальных признаков structured-таблицы. Фактовая таблица содержит ID справочников и числовые показатели.

При загрузке фактов применяется join structured-данных со справочниками по полю name, после чего в факт записываются суррогатные идентификаторы. Такой подход снижает избыточность данных и делает модель более удобной для аналитических запросов, так как фильтрация и агрегация выполняются по компактным ключам.

4.2.3 Формирование витрины данных

Создано представление v_dm_task с явным перечислением полей без использования select *.

4.2.4 Перенос витрины в целевую СУБД

Реализована функция Python, которая переносит данные из PostgreSQL-витрины в таблицу MySQL sql_dm.t_dm_task.

5 Инструкция по запуску проекта

5.1 Подготовка окружения

Необходимые инструменты: Docker Desktop, DBeaver.

Запуск БД-контейнеров:

docker compose up -d postgres mysql

5.2 Запуск ETL-процесса

docker compose run --rm app --step etl

5.3 Формирование витрины данных

docker compose run --rm app --step dm

5.4 Перекладка в MySQL

docker compose run --rm app --step transfer

Либо одним шагом:

docker compose run --rm app --step all

ЗАКЛЮЧЕНИЕ

В ходе выполнения лабораторных работ реализован работоспособный конвейер обработки данных: от генерации сырых данных до построения витрины и переноса в целевую СУБД. Все ключевые требования ЛР1 и ЛР2 соблюдены: SQL-функции параметризованы периодом, реализована верхнеуровневая функция ETL без входных параметров, создана DWH-структура со справочниками и таблицей фактов, сформировано представление-витрина с явным списком полей, выполнена перекладка данных в MySQL.
